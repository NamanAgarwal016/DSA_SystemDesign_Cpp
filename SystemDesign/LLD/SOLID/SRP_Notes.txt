Detailed points, why they matter, and corrected code
1. Use member initializer list in constructors
Why: initializer lists initialize members directly and can be more efficient and correct for non-trivial types.
Your code:
Product(string name, double price) {
    this->name = name;
    this->price = price;
}
Better:
Product(string name, double price)
    : name(std::move(name)), price(price) { }
Notes:

std::move(name) avoids an extra copy if the caller passes an rvalue.
For small types like string it’s still good practice to use initializer lists.


2. Avoid copying where unnecessary — pass by const reference / iterate by const reference
Why: copying objects in loops or function parameters can be costly and unnecessary.
Your code:
void addProduct(Product product) {
    products.push_back(product);
}

for (Product p : products) { ... }
Better:
void addProduct(const Product& product) {
    products.push_back(product); // copy only once
}
// or the take-by-value-and-move pattern:
void addProduct(Product product) {
    products.push_back(std::move(product));
}

for (const Product& p : products) {
    // use p without copying
}

Choose:

const Product& when you want to avoid copying and don't need ownership transfer.
"Take by value and move" when callers may pass rvalues often and you want to optimize both lvalues and rvalues.


3. Mark getters and read-only methods as const
Why: Improves correctness and allows calling these methods on const objects.
Your code:
vector<Product> getProducts() {
    return products;
}
double calculateTotalPrice(){
    ...
}
Better:
const vector<Product>& getProducts() const {
    return products;
}

double calculateTotalPrice() const {
    ...
}
Notes:

Returning const vector<Product>& avoids copying the entire container.
Marking the method const documents that it doesn't mutate the object.
Return containers by reference when appropriate
Why: copy of vector can be expensive. Returning a reference (const) avoids copying.
Your code returns vector<Product> by value. Prefer:
If you need to return an owned container (that callers will modify independently), returning by value is correct — but not here.


5. Iterate with auto / const auto& for clarity and correctness
Why: if you change container element type to pointer or smart pointer later, auto avoids mistakes.
Prefer:
for (const auto& p : products) {
    cout << p.name << ": " << p.price << '\n';
}

Suggested corrected version (keeps value semantics, more idiomatic C++)
This keeps your value semantics but applies the points above.

#include <iostream>
#include <string>
#include <vector>

using namespace std;

struct Product {
    string name;
    double price;

    Product(string name, double price)
        : name(std::move(name)), price(price) {}
};

class ShoppingCart {
private:
    vector<Product> products;

public:
    // Take by const-ref to avoid unnecessary copy
    void addProduct(const Product& product) {
        products.push_back(product);
    }

    // Return const ref; method is const
    const vector<Product>& getProducts() const {
        return products;
    }

    double calculateTotalPrice() const {
        double total = 0.0;
        for (const auto& p : products) {
            total += p.price;
        }
        return total;
    }

    void printInvoice() const {
        cout << "Shopping Cart Invoice:\n";
        for (const auto& p : products) {
            cout << p.name << " - Rs " << p.price << '\n';
        }
        cout << "Total: Rs " << calculateTotalPrice() << '\n';
    }

    // Persistence responsibility should be extracted in real code.
    void saveToDB() const {
        cout << "Saving to DB....\n";
    }
};

int main() {
    ShoppingCart cart;
    cart.addProduct(Product{"Car", 20.00});
    cart.addProduct(Product{"Bike", 10.00});
    cart.printInvoice();
    cart.saveToDB();
    return 0;
}

Why this is good:

No unnecessary copying while still using simple value semantics.
Methods that don't change state are const.
Uses member initializer list when constructing Product.
Clear separation candidate: the saveToDB() method is marked const but you should extract it to a repository class later.

Final notes — when to use pointers vs objects (short)
Prefer value semantics (objects) for simple, copyable types and when ownership is local.
Use pointers (smart pointers) when you need:
polymorphism (base* to derived),
ownership transfer or shared ownership,
objects that must outlive the creating scope.
When using pointers, prefer std::unique_ptr or std::shared_ptr over raw new/delete.

4) InvoicePrinter / DBManager store cart by value — unwanted copying
Your classes:
class InvoicePrinter {
private:
    ShoppingCart cart;
public:
    InvoicePrinter(ShoppingCart& cart) : cart(cart) {}
    ...
};
What's wrong:

The constructor takes a reference but the class stores a copy (by-value member). That copies the entire ShoppingCart. This means the printer prints the snapshot at construction time, not the live cart. Also copying can be expensive. Probably you intended to keep a reference.
Fix: store a reference (or const reference) instead:

class InvoicePrinter {
private:
    const ShoppingCart& cart;   // reference to the real cart
public:
    InvoicePrinter(const ShoppingCart& cart) : cart(cart) {}

    void printInvoice() const {
        for (const auto& p : cart.getProducts()) {
            cout << p.name << ": " << p.price << '\n';
        }
        cout << "Total: " << cart.calculateTotal() << '\n';
    }
};
